## Detailed Issue Breakdown

### Scope Confusion (A1) — CRITICAL

**Problem**: Feature titled "Code Block Preview Background Refinement" but spec includes two unrelated feature areas:
1. **US1-US2**: Screenshot/preview pane background theming (true background refinement)
2. **US4**: Code editor gutter change indicators (independent developer feature with no connection to backgrounds)

**Impact**: Users unfamiliar with the spec may expect a single cohesive change. Implementation phases are interleaved without clear justification. Tasks 001-007 feel like finishing an incomplete prior feature, while 008-024 introduce a new feature.

**Evidence**:
- spec.md feature title: "Code Block Preview Background Refinement" (singular feature)
- spec.md content: 3 user stories with distinct concerns (backgrounds, separation, indicators)
- plan.md summary: Different phrasing ("enhance the code block editing experience")
- tasks.md section headers: Organized by user story (US1, US2, US4 but separately)

**Recommendation**: Either (a) rename feature to "Code Editor & Snapshot Preview UX Refinement" to encompass all three stories, or (b) split into two specs: `011-snapshot-theme-refinement` (US1-US2) and `012-change-indicators` (US4). The current state is ambiguous.

### Design Alternative Ambiguity (AM1) — CRITICAL

**Problem**: Spec proposes three design alternatives (A: theme-aware solid, B: transparent, C: gradient) with clear trade-offs. Spec states "implementation should adopt one during planning/clarification." Plan.md immediately states "Already implemented: Alternative B."

**Impact**: No evidence of design decision. No artifact documenting which alternative chosen or why rejected others. If stakeholder disagrees with Alternative B, no recourse in plan. If Alternative B implementation is incomplete, no backup plan.

**Evidence**:
- spec.md "Design Alternatives" section: "implementation should adopt **one** of them (decided during planning/clarification)"
- plan.md Summary: "Alternative B — transparent container with theme-aware modal **already implemented**"
- tasks.md: No task for design decision; T001-T004 assume Alternative B

**Recommendation**: Create design-decision.md artifact documenting: (1) alternatives considered (A, B, C), (2) selected alternative (B) with trade-offs, (3) rejected alternatives with reasoning, (4) stakeholder approval date. Add task T0.0 for this decision phase. Reference from plan.md.
